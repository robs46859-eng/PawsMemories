# Blender Microservice Troubleshooting Guide

This guide covers the root causes of the rendering and rigging errors encountered in the `blender-worker` microservice, how they relate to the Blender version mismatch, and steps to resolve them.

## 1. Why is the build using Blender 3 instead of Blender 5?

Your `blender-worker` uses a Dockerfile based on `node:20-slim` (which uses Debian Linux). In your `Dockerfile`, Blender is installed via the system package manager:
```dockerfile
RUN apt-get update && apt-get install -y blender
```
Debian's stable repositories are notoriously conservative to prioritize system stability. The version of Blender shipped in the default Debian Bookworm repositories is **Blender 3.4.1**. 

Because you are using `apt-get install blender`, Docker downloads the older 3.4.1 package instead of the newer Blender 5.1.

### How to Upgrade to Blender 5.x
If you want to use Blender 5 (which matches the `blender_manual_v510` docs you downloaded), you cannot use `apt-get`. Instead, you must download the official portable Linux tarball directly from the Blender Foundation in your Dockerfile:

```dockerfile
# Example of installing a specific newer Blender version in Docker
RUN apt-get update && apt-get install -y wget xz-utils libegl1 libxrender1 libxi6 && \
    wget https://download.blender.org/release/Blender5.1/blender-5.1.0-linux-x64.tar.xz && \
    tar -xvf blender-5.1.0-linux-x64.tar.xz -C /usr/local/ && \
    ln -s /usr/local/blender-5.1.0-linux-x64/blender /usr/bin/blender && \
    rm blender-5.1.0-linux-x64.tar.xz
```

## 2. API Mismatch Errors (Hallucinations vs. Version Changes)

Because you have been referencing Blender 5.1 workflows or AI models trained on newer documentation, the AI generated Python scripts containing APIs that either do not exist in Blender 3.4.1 or were entirely hallucinated.

### Error: `AttributeError: 'NlaStrip' object has no attribute 'is_active'`
* **Cause**: The AI hallucinated a property (`is_active`) to disable NLA strips.
* **Fix**: Use `strip.mute = True`. This has been hardcoded into the post-processing script.

### Error: `TypeError: property "pose.bones["neck"].rotation_euler" not found`
* **Cause**: In Blender Python, when calling `.keyframe_insert()` directly on a pose bone, the `data_path` must be relative to the bone (e.g., `"rotation_euler"`). Passing the absolute path from the armature (`"pose.bones['neck'].rotation_euler"`) causes a type error.
* **Fix**: The AI prompt has been updated with explicit rules to use relative data paths.

### Error: `'NoneType' object has no attribute 'rotation_euler'`
* **Cause**: The AI tried to animate a bone that did not exist in the rigged GLB generated by HuggingFace. `pose.bones.get("bone_name")` returned `None`, which crashed the script.
* **Fix**: Added defensive programming rules (`if bone is not None:`) to the generation prompt so missing bones are safely ignored.

## 3. Worker Environment & Export Warnings

### Warning: `ERROR: Draco mesh compression is not available`
* **Cause**: Blender 3.4's glTF exporter looks for the Draco compression library (`libextern_draco.so`). The default Debian package for Blender often strips out proprietary or external compression libraries to comply with open-source packaging rules.
* **Impact**: Safe to ignore. The glTF files will still export perfectly fine, they just won't be compressed with Draco (they will be slightly larger in file size).

### Error: `BadRequestError: request aborted`
* **Cause**: The `blender-worker` was receiving 15MB+ base64-encoded GLB files. While Express was parsing this massive JSON payload, the client connection timed out or dropped. Express threw an unhandled error, crashing the worker.
* **Fix**: Added a global error-handling middleware in `server.js` to gracefully catch and reject `request.aborted` and `entity.too.large` exceptions without crashing the main process.

### Error: `could not get a list of mounted file-systems`
* **Cause**: A harmless warning emitted by Blender when running inside a heavily sandboxed Docker container or headless cloud instance (like Render.com) where it lacks permissions to scan the host OS's mounted drives.
* **Impact**: Completely harmless. Can be ignored.

## Summary & Next Steps

Currently, the AI generation pipeline has been deeply sanitized to be compatible with **Blender 3.4.1**. If you choose to upgrade the Docker container to Blender 5 using the tarball method described above, you may need to update the AI prompt rules (specifically removing the `TARGET BLENDER VERSION is 3.4` constraint) to allow the AI to utilize newer Blender 5 features like Geometry Nodes or updated EEVEE capabilities.
